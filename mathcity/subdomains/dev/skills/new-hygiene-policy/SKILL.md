# new-hygiene-policy

Propose and commit an amendment to `mathcity/subdomains/dev/POLICY.md` —
the source-of-truth P-rules document that `check-plan-hygiene` enforces.
Every policy change is gated on Taylor approval and recorded in a Change Log
section before it becomes enforceable.

**Companion:** [[check-plan-hygiene]] — run after any amendment to confirm
the new P-rule is detectable and no plan already in flight violates it.

## When to use

- `check-plan-hygiene` returns a finding with no matching P-rule
- A vocabulary or terminology rule must be enforced across all plans and skill docs
- A privacy, ownership, or reproducibility gap is identified after an incident
- A new constraint is added to gascity substrate (new pack family, new rig type,
  new deployment target) and POLICY.md doesn't cover it
- An existing P-rule is wrong, ambiguous, or creates false positives

## Step 0 — Read current policy

```bash
cat ~/repos/gascity-packs/mathcity/subdomains/dev/POLICY.md
```

Note the latest P-rule ID, the **Date** line in the header table, and
the existing Pillars. New P-rules increment the highest existing ID within
their Pillar (P1.x, P2.x, P3.x, P4.x) or open a new Pillar (P5.x) if the
rule doesn't fit any existing pillar.

## Step 1 — Draft the amendment

State the amendment as a structured proposal:

```
PROPOSED AMENDMENT — <date>

Pillar: <1=Reproducibility | 2=Ownership | 3=Upstream | 4=Impact | 5=new-pillar>
Rule ID: <P<N>.<M> — next available in the pillar>
Rule title: <short name, e.g. "No gastown vocabulary in plan prose">

Trigger: <what prompted this — incident, check-plan-hygiene finding, Taylor directive>

Proposed rule text:
  <The P-rule in the same style as existing POLICY.md rules — one paragraph,
   with clear pass/fail criterion. State what the rule prohibits, the allowed
   exceptions, and the verdict (fail / revise / defer) when violated.>

Allowed exceptions (if any):
  <Explicit carve-outs — e.g. "gastown.* is a runtime identity namespace and
   must not be renamed; its literal appearance in agent config is allowed.">

Pass criterion (for check-plan-hygiene to enforce):
  <One sentence on what a checker looks for to PASS: "No occurrence of X in
   plan prose outside of Y.">

Fail criterion:
  <One sentence on what triggers a FAIL: "Any plan doc or skill prose using
   word X in context Y triggers P<N>.<M>.">

Downstream impact:
  - Plans/skills currently in violation of the NEW rule: <list or "none">
  - Remediation required: <yes/no; if yes, describe>
```

Present the draft to Taylor before touching the file. Do NOT edit POLICY.md
until Taylor approves.

## Step 2 — Gate on Taylor approval

Use [[present-it]] compact form for straightforward vocabulary/terminology rules:

```
DECISION:  Add P<N>.<M> ("<rule title>") to mathcity/subdomains/dev/POLICY.md?
CONTEXT:   <one sentence on what triggered this>
RECOMMEND: APPROVE — <one-line rationale>
CONFIRM:   y / n / grill-me-further
```

For rules that change a Pillar's scope, add a new Pillar, or affect existing
plans in flight: use full-form [[present-it]] with all seven sections.

**Taylor's approval in this conversation is required.**

## Step 3 — Apply the amendment

After Taylor approves, edit `~/repos/gascity-packs/mathcity/subdomains/dev/POLICY.md`:

1. Add the new rule under the correct Pillar section, maintaining numeric order.
   Use the same format as existing P-rules (bold rule ID, pass/fail criterion inline).
2. Update the **Date** line in the header table to today.
3. Add or extend the Change Log at the end of the file:

```markdown
---

## Change Log

### <date> — P<N>.<M> added: <rule title>
<One sentence summary>. Triggered by: <incident or directive>.
Exceptions: <list or "none">.
```

4. Run gitleaks on the amended file:
```bash
gitleaks detect --no-git \
  --source ~/repos/gascity-packs/mathcity/subdomains/dev/ 2>&1 | grep -v "no leaks"
```

5. Commit and push to the fork (standing authorization gt-5cye):
```bash
cd ~/repos/gascity-packs
git add mathcity/subdomains/dev/POLICY.md
git commit -m "policy: add P<N>.<M> — <rule title> [autogenerated by <model> on <date>]"
git push fork main
```

NEVER push to `gastownhall/gascity-packs` upstream. Fork only.

## Step 4 — Verify with check-plan-hygiene

After amendment, run [[check-plan-hygiene]] on a representative plan to confirm:
- The new rule is detectable (the check procedure can cite the new P-rule ID)
- No plan currently in flight fails the new rule unexpectedly
- POLICY.md parses correctly (no broken markdown, no orphaned header)

## Hard rules

- **Never edit POLICY.md without Taylor approval** (Step 2 mandatory).
- **Never change a P-rule to excuse an existing violation** — fix the violation,
  then optionally amend to prevent recurrence.
- **Never push to gastownhall/gascity-packs upstream** — fork only (gt-5cye).
- **P-rule IDs are permanent** — once assigned, a P-rule ID is never reused,
  even if the rule is later repealed. Repeal = mark "REPEALED: <date>" in the
  rule body, not deletion.
- **Allowed exceptions must be precise** — vague exceptions ("unless it makes
  sense") defeat the rule. Every exception states exactly what is allowed and why.
- **Check-plan-hygiene is the enforcer** — if the new rule can't be expressed
  as a detectable pass/fail criterion, it's not a policy rule; it's guidance.
  Guidance goes in a separate doc, not POLICY.md.
